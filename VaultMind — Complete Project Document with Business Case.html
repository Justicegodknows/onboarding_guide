⚠️ Risk Assessment
Key risks and mitigations for SME deployment
HIGH
Hardware Procurement Delays
GPU shortages can delay delivery past Week 1. Use RunPod or Lambda Labs cloud GPU as temporary dev environment — switch to on-premise for production go-live.
→ Order hardware Day 1 of project. Cloud GPU fallback pre-configured and ready.
HIGH
Answer Hallucination Risk
LLM may generate plausible but incorrect answers — critical risk for compliance-sensitive SMEs in legal, finance, or healthcare.
→ Strict RAG-only mode enforced. Source citations mandatory on every response. Confidence scoring + UAT before launch.
MEDIUM
Employee Adoption Resistance
Staff may default to old habits — emailing colleagues instead of querying VaultMind.
→ Pilot with 5–10 champions first. Gather testimonials. Live demo at company all-hands. Simple one-page onboarding guide.
LOW
Model Licensing Changes
Open-source model providers could change commercial licensing terms.
→ Architecture is fully model-agnostic via Ollama. Any compatible model swapped in with zero code changes.

🛠️ Tech Stack
One tool per layer — all open-source, 100% on-premise, zero licensing cost, zero cloud dependency
🖥️ Layer 1 — LLM Inference Engine: Ollama
ToolRole & RationaleLink
Ollama (via Docker)Runs all LLM models locally. Single-command setup, GPU-accelerated, OpenAI-compatible REST API. Acts as the unified runtime for both LLM inference and embedding generation. Zero cloud calls — all inference is local.ollama.com
🤖 Layer 2 — Base LLM Model: Selected via Ollama Library at Setup
Selection MethodRole & RationaleLink
Ollama Model LibraryNo model hardcoded. The optimal model is pulled from Ollama's library at deployment time based on available GPU VRAM and performance testing — choose from Llama 3, Mistral, Qwen2.5, DeepSeek-R1. Fully swappable with zero code changes.ollama.com/library
⚙️ Layer 3 — RAG Framework: LangChain
ToolRole & RationaleLink
LangChainOrchestrates the full RAG pipeline — document loaders, text chunking, embedding generation, vector store retrieval, prompt construction, and LLM response formatting. Industry-standard, battle-tested, extensive connector library.langchain.com
🗄️ Layer 4 — Vector Database: ChromaDB
ToolRole & RationaleLink
ChromaDBStores and queries document embeddings locally. Lightweight, fully self-hosted, Python-native, no server infrastructure required. Ideal for SME document libraries (up to millions of chunks). Integrates natively with LangChain.trychroma.com
📄 Layer 5 — Document Parsing: PyMuPDF (fitz)
ToolRole & RationaleLink
PyMuPDF (fitz)Extracts text, tables, and metadata from PDFs with high accuracy and speed. Handles multi-page, scanned, and complex formatted documents. Lightweight Python library — no external service required. Also handles DOCX and image-based PDFs when combined with pytesseract.pymupdf.readthedocs.io
🔢 Layer 6 — Embedding Models: HuggingFace + Ollama
ToolRole & RationaleLink
HuggingFace sentence-transformers + Ollama nomic-embed-textConverts document chunks and user queries into vector embeddings for semantic similarity search. HuggingFace provides access to state-of-the-art models (BGE-M3, all-MiniLM). Ollama's nomic-embed-text runs locally with zero API cost. Both options run entirely on-premise.sbert.net · Ollama Embed
🖥️ Layer 7 — Frontend UI: Streamlit
ToolRole & RationaleLink
StreamlitBuilds the employee-facing chat interface in pure Python — no frontend development required. Renders chat history, source citations, document previews, and admin controls. Runs as a Docker container on the local network, accessible via browser.streamlit.io
🐍 Layer 8 — Backend API: Python / FastAPI
ToolRole & RationaleLink
Python + FastAPIHandles all backend logic — query routing, authentication middleware, document ingestion pipeline, ChromaDB interactions, Ollama API calls, and audit logging. FastAPI provides async REST endpoints with automatic OpenAPI documentation. Fast, lightweight, production-ready.fastapi.tiangolo.com
🔑 Layer 9 — Security & Access Control: JWT Tokens
ToolRole & RationaleLink
JWT (JSON Web Tokens)Enforces role-based access control — each user receives a signed JWT containing department and permission claims. FastAPI middleware validates tokens on every request. Ensures HR staff only retrieves HR documents, Finance only retrieves Finance documents. Stateless, scalable, industry-standard.jwt.io
🐳 Layer 10 — Deployment & Infrastructure: Docker + LocalStack
ToolRole & RationaleLink
Docker + Docker Compose + LocalStackAll services (Ollama, FastAPI, ChromaDB, Streamlit) run as Docker containers orchestrated by Docker Compose. LocalStack emulates AWS services locally (S3, SQS) for document ingestion pipelines — without any actual cloud dependency. Single docker-compose up command deploys the entire system.docker.com · localstack.cloud

Employee Opens VaultMind

Employee navigates to VaultMind on their browser (local intranet URL). The Streamlit interface loads — a clean chat UI similar to ChatGPT, hosted entirely on your internal network. No internet connection required.
Tool: Streamlit · Docker container · Accessible via http://vaultmind.local or internal IP
↓
2
🔑
Login & JWT Token Issued
JWT Auth
Employee logs in with company credentials. FastAPI backend validates credentials and issues a signed JWT token containing the user's department, role, and access permissions. All subsequent requests carry this token — no re-authentication needed per session.
Tool: Python FastAPI + python-jose JWT · Token contains: user_id, department, permissions, expiry

💬
Employee Types a Question
Streamlit → FastAPI
Employee types a natural language question (e.g. "What is our parental leave policy?" or "What were Q3 2025 revenue figures?"). Streamlit sends the query + JWT token to the FastAPI backend via a secure POST request on the local network.
Tool: Streamlit chat_input → HTTP POST to FastAPI /query endpoint · JWT validated on every request
↓
4
🔢
Query Converted to Vector Embedding
HuggingFace / Ollama Embed
FastAPI passes the user's question to the embedding model (HuggingFace sentence-transformer or Ollama nomic-embed-text). The model converts the text into a high-dimensional vector — a numerical representation of the question's semantic meaning.
Tool: sentence-transformers (all-MiniLM-L6-v2) or nomic-embed-text via Ollama · Runs 100% locally · 384–768 dimensional vector output
↓
5
🗄️
Semantic Search in ChromaDB
ChromaDB
The query vector is compared against all stored document chunk vectors in ChromaDB using cosine similarity. The top 3–5 most semantically relevant document chunks are retrieved. Access is filtered by the user's JWT department claim — a Finance user only searches Finance collections.
Tool: ChromaDB collection.query() · Filters applied: department = JWT claim · Returns: top-k chunks + source metadata
↓
6
⚙️
LangChain Builds the Prompt
LangChain RAG
LangChain constructs a structured prompt combining: (1) the retrieved document chunks as context, (2) the user's original question, and (3) system instructions enforcing RAG-only mode — the LLM is instructed to answer only from the provided context and to say "I don't know" if the answer isn't present.
Tool: LangChain RetrievalQA chain · Prompt template enforces: context-grounded answers only · No hallucination outside retrieved docs
↓
7
🤖
Ollama LLM Generates the Answer
Ollama (Local LLM)
The prompt is sent to Ollama's local REST API. The selected LLM (e.g. Llama 3 8B, Mistral 7B) processes the prompt entirely on your GPU — no internet required. The model generates a precise, context-grounded answer in natural language. Zero tokens sent to OpenAI or any external API.
Tool: Ollama REST API (localhost:11434) · GPU-accelerated inference · Avg. response time: 3–8 seconds on RTX 4090
↓
8
📝
Audit Log Written
FastAPI + SQLite/Postgres
Before returning the response, FastAPI writes a complete audit record: user ID, department, exact query text, documents retrieved (filenames + page numbers), timestamp, and response latency. This log satisfies GDPR Article 30 (Records of Processing Activities) and ISO 27001 requirements.
Tool: FastAPI background task → SQLite or Postgres · Fields: user_id, query, sources[], timestamp, latency_ms

✅

Streamlit UI
The answer is streamed back to the Streamlit UI and displayed in the chat window. Below the answer, source citation cards show: document name, page number, and the exact extracted passage used. The employee can trust the answer — it came directly from a company document they have permission to access.
Tool: Streamlit chat_message + expander cards for citations · Total round-trip: 4–10 seconds · Zero data left the building

At no point in this 9-step flow does any data leave your servers. The LLM runs locally via Ollama. The vector database is local ChromaDB. The embeddings are generated locally. The UI is served locally. Your documents, your queries, and your answers never touch the internet.
🔄 RAG Pipeline — Detailed Flow
How Retrieval-Augmented Generation works inside VaultMind — document ingestion and query-time retrieval
What is RAG? Retrieval-Augmented Generation is a technique where the LLM does NOT answer from its training data alone. Instead, it first retrieves the most relevant passages from your actual company documents, then generates an answer grounded in those passages. This eliminates hallucination and ensures every answer is traceable to a real source.
📥 Phase 1 — Document Ingestion (One-Time Setup + Ongoing)
1
📁
Document Source
Admin uploads documents via Streamlit UI or drops files into a watched folder
Supports: PDF, DOCX, XLSX, PPTX, TXT, images — all document types ingested automatically
2
📄
Document Parsing
PyMuPDF (fitz) extracts raw text, tables, and metadata from each document
Preserves: page numbers, headings, table structure, author, date — all stored as metadata
3
✂️
Text Chunking
LangChain splits documents into overlapping chunks of 500–1000 tokens
Overlap of 50–100 tokens preserves context across chunk boundaries — no information lost
4
🔢
Embedding Generation
Each chunk converted to a vector embedding via HuggingFace or Ollama nomic-embed-text
Vector captures semantic meaning — similar concepts cluster together, enabling semantic search
5
🗄️
Vector Storage
Embeddings + metadata stored in ChromaDB with department-level collection tagging
Each chunk tagged with: source file, page, department, date ingested — enables filtered retrieval
🔍 Phase 2 — Query-Time Retrieval & Generation (Every User Question)
6
💬
User Query
Employee types question in Streamlit → FastAPI receives query + JWT token
JWT decoded to extract department permissions — filters applied to ChromaDB search scope
7
🔢
Query Embedding
User's question embedded into vector using same embedding model as ingestion
Critical: same model must be used for both ingestion and query — ensures vector space consistency
8
🎯
Semantic Retrieval
ChromaDB finds top-k chunks with highest cosine similarity to query vector
k=3–5 chunks retrieved · Filtered by department · Returns chunk text + source metadata
9
📝
Prompt Construction
LangChain assembles: system instructions + retrieved context chunks + user question
System prompt enforces: "Answer ONLY from the provided context. If unsure, say I don't know."
10
🤖
LLM Generation
Ollama LLM generates natural language answer grounded in retrieved context
Model reads context, synthesises a clear answer in plain English — does NOT use training knowledge outside context
11
✅
Cited Response Delivered
Answer + source citations (document name, page, passage) returned to Streamlit UI
Employee sees: answer + "Source: HR Policy Manual, Page 12" — fully auditable and trustworthy
